06. Visão Futura — Ecossistema IX

Rafael mencionou a possibilidade de criar o próprio ecossistema da Impact X — métricas, documentações, relatórios, gestão de projetos, tudo com a metodologia DOE. O que já existe foi analisado junto ao gap real.

O que já existe

Componente Status Onde vive
Publicação de relatórios Operacional ix-reports (reports.impactxlab.com)
Gestão de projetos Operacional Plane (cloud, free tier)
Monitoramento de agentes Operacional Paperclip Dashboard
Automações determinísticas Operacional ~/agent-workspace/scripts/ (30+ scripts)
Integrações Slack/Notion/HubSpot Operacional MCPs oficiais
CI/CD de agentes Parcial dispatch.sh + run.sh

O gap real

Insight Não há uma camada unificada de métricas de negócio. Plane tem tasks, HubSpot tem pipeline, Paperclip tem custos de AI, ix-reports tem documentos. Nenhum consolida. O gap estrutural é um relatório de KPIs consolidado — MRR, pipeline, custos, tarefas abertas — não um sistema novo.

Plane Self-Hosted na VPS Oracle

O Plane é open-source e o deploy é via Docker Compose. A VPS Oracle (ARM, 4 OCPU, 24GB RAM) aguenta sem problema. Mas o custo oculto é operacional: updates, backups, SSL, monitoramento de uptime. Sem pipeline de manutenção automatizada, se a instância cair o PM do time cai junto.

P1 Recomendação: manter Plane cloud por enquanto. Self-host quando houver pipeline de manutenção na VPS (requer ~1 sprint separado).

Arquitetura proposta do ecossistema (3 componentes)

ix-metrics (novo)

Script semanal que puxa Plane + HubSpot + Paperclip e gera um relatório HTML no formato ix-reports. Não é dashboard complexo — é relatório vivo.

VPS como plataforma de agentes

Mover agentes pesados (cron) para a VPS, liberar o Mac do Rafael. Falta pipeline de deploy + monitoramento de uptime com alerta no Slack.

Plane como registro único

Padronizar: Plane = fonte de verdade de tasks. Paperclip = apenas custo/métricas de AI. Execuções markdown = arquivo histórico.

Trade-off: construir vs. compor

Atenção A tentação é construir um painel bonito. O risco é criar infraestrutura que precisa de manutenção quando o objetivo é menos tempo gerenciando ferramentas. Se uma ferramenta existente resolve 80% com composição (MCP + script + ix-reports), não construir. Só construir quando o gap é estrutural e recorrente.

Próxima fase (quando houver sprint disponível)

P0 ix-metrics: script semanal de KPIs consolidados (impacto imediato, baixo risco)

P1 Pipeline de deploy na VPS: automatizar updates e monitoramento

P2 Plane self-hosted: só após pipeline de manutenção da VPS

P2 Dashboard interativo (Grafana/Metabase): overhead supera benefício para time de 4 pessoas
